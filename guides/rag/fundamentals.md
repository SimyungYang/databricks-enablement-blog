# RAG 핵심 개념 사전

> 이 페이지는 RAG를 처음 접하는 분을 위해, 등장하는 모든 알고리즘과 개념을 **왜 필요한지**, **무엇을 하는지**, **쉬운 비유**와 함께 설명합니다. AI나 NLP 배경지식이 없어도 이해할 수 있도록 작성했습니다.

---

## 1. 임베딩 (Embedding) — "문장을 숫자로 바꾸기"

### 왜 필요한가?

컴퓨터는 "오늘 날씨 좋다"와 "오늘 하늘이 맑다"가 비슷한 의미라는 것을 모릅니다. 컴퓨터에게 텍스트는 그저 문자 코드의 나열일 뿐이기 때문입니다. **의미를 비교**하려면, 텍스트를 컴퓨터가 계산할 수 있는 **숫자 형태**로 바꿔야 합니다.

### 무엇인가?

임베딩은 텍스트를 수백~수천 개의 숫자로 이루어진 배열(벡터)로 변환하는 기술입니다.

```
"오늘 날씨가 좋다" → [0.12, -0.34, 0.56, 0.78, -0.11, ...]  (1536개 숫자)
"오늘 하늘이 맑다" → [0.13, -0.32, 0.54, 0.80, -0.09, ...]  (1536개 숫자)
"주식 시장이 폭락했다" → [-0.45, 0.67, -0.23, 0.11, 0.89, ...]  (1536개 숫자)
```

의미가 비슷한 문장은 숫자 배열도 비슷해지고, 의미가 다른 문장은 숫자 배열도 크게 달라집니다.

### 비유

도서관에서 모든 책에 **주제 좌표**를 매기는 것과 같습니다. "요리책"과 "레시피 모음"은 가까운 좌표에, "요리책"과 "양자역학 교과서"는 먼 좌표에 배치됩니다.

### 주요 임베딩 모델

| 모델 | 제공자 | 차원 수 | 특징 |
|------|--------|---------|------|
| text-embedding-3-small/large | OpenAI | 1536/3072 | 가장 널리 사용 |
| BGE (BAAI General Embedding) | BAAI | 768/1024 | 오픈소스, 다국어 지원 |
| E5 | Microsoft | 768/1024 | 오픈소스, 높은 성능 |
| GTE (General Text Embeddings) | Alibaba | 768/1024 | 오픈소스, 다국어 |

{% hint style="info" %}
**Databricks 활용**: Foundation Model API를 통해 다양한 임베딩 모델을 Serving Endpoint로 호출할 수 있습니다. Unity Catalog의 AI Functions(`ai_query`)를 사용하면 SQL만으로도 임베딩을 생성할 수 있습니다.
{% endhint %}

## 2. 벡터 (Vector) & 벡터 공간

### 벡터란?

임베딩의 결과물이 바로 **벡터**입니다. 수학적으로는 방향과 크기를 가진 숫자 배열입니다.

```
벡터 예시: [0.12, -0.34, 0.56]  ← 3차원 벡터
실제 임베딩: [0.12, -0.34, 0.56, ..., 0.78]  ← 1536차원 벡터
```

### 벡터 공간이란?

모든 문서의 벡터가 배치된 **다차원 공간**입니다. 우리가 사는 세상은 3차원이지만, 임베딩 벡터는 수백~수천 차원의 공간에 존재합니다.

### 비유

세계 지도 위에 도시들을 점으로 찍는 것과 같습니다. 서울과 부산은 가깝고, 서울과 뉴욕은 멉니다. 벡터 공간에서도 마찬가지로, **의미가 가까운 문서는 가까운 위치에**, 의미가 먼 문서는 먼 위치에 놓입니다.

## 3. 유사도 측정 — "두 문장이 얼마나 비슷한가?"

두 벡터가 있을 때, 이들이 얼마나 비슷한지 측정하는 방법이 여러 가지 있습니다.

### Cosine Similarity (코사인 유사도)

두 벡터가 이루는 **각도**를 측정합니다.

- **범위**: -1 ~ 1 (보통 0 ~ 1로 정규화)
- **1에 가까울수록** 의미가 비슷
- **0에 가까울수록** 관련 없음
- RAG에서 **가장 많이 사용**되는 방법

```
cosine_similarity("오늘 날씨 좋다", "하늘이 맑다") = 0.92  ← 매우 비슷
cosine_similarity("오늘 날씨 좋다", "주식이 폭락했다") = 0.15  ← 관련 없음
```

### Euclidean Distance (유클리드 거리)

두 벡터 사이의 **직선 거리**를 측정합니다. 지도에서 두 점 사이의 거리를 자로 재는 것과 같습니다.

- **거리가 짧을수록** 비슷
- 벡터의 크기(길이)에 영향을 받음

### Dot Product (내적)

두 벡터의 **크기와 방향**을 모두 고려하는 연산입니다.

- 값이 **클수록** 비슷
- 벡터의 크기가 정규화되어 있으면 코사인 유사도와 동일한 결과

{% hint style="info" %}
**실무 팁**: 대부분의 임베딩 모델은 벡터를 정규화하여 출력하므로, 코사인 유사도와 내적의 결과가 같습니다. Databricks Vector Search에서는 기본적으로 코사인 유사도를 사용합니다.
{% endhint %}

## 4. 벡터 인덱스 — "수백만 개에서 빠르게 검색하기"

### 왜 필요한가?

100만 개의 문서가 있을 때, 질문과 가장 비슷한 5개를 찾으려면 100만 번의 유사도 계산이 필요합니다. 이렇게 하면 **너무 느립니다**. 벡터 인덱스는 이 검색을 극적으로 빠르게 만드는 자료 구조입니다.

### ANN (Approximate Nearest Neighbor)

**정확한** 가장 가까운 이웃을 찾는 대신, **대략적으로** 가장 가까운 이웃을 찾습니다. 정확도를 약간 희생하는 대신 속도를 수십~수백 배 향상시킵니다.

**비유**: 도서관에서 원하는 책을 찾을 때, 모든 책을 하나씩 확인하는 대신 먼저 **서가 분류**(문학, 과학, 역사...)를 확인하고 해당 서가에서만 찾는 것입니다.

### HNSW (Hierarchical Navigable Small World)

ANN의 대표적인 알고리즘입니다. 데이터를 **계층적 그래프 구조**로 연결하여 빠르게 탐색합니다.

**작동 원리** (단순화):
1. 데이터를 여러 계층(레이어)으로 나눔
2. 최상위 계층에서 대략적인 위치를 찾음 (고속도로로 대략 이동)
3. 하위 계층으로 내려가며 점점 정밀하게 탐색 (골목길로 정확히 이동)

**비유**: 네비게이션이 경로를 찾는 방식과 같습니다. 먼저 고속도로로 대략적인 방향을 잡고, 목적지 근처에서 세부 도로로 들어갑니다.

{% hint style="info" %}
**Databricks Vector Search**는 내부적으로 HNSW 알고리즘을 사용합니다. 사용자는 알고리즘 세부 사항을 신경 쓸 필요 없이 Delta 테이블만 연결하면 자동으로 인덱스가 생성됩니다.
{% endhint %}

## 5. BM25 (Best Matching 25) — "키워드 매칭의 왕"

### 왜 필요한가?

임베딩 기반 검색은 **의미**를 잘 이해하지만, 정확한 **키워드 매칭**에는 약할 수 있습니다. 예를 들어, 제품번호 "ABC-123"이나 오류 코드 "ERR_CONNECTION_REFUSED" 같은 고유한 문자열은 임베딩으로 찾기 어렵습니다.

### 무엇인가?

BM25는 문서에서 검색어가 **얼마나 자주**, **얼마나 드물게** 등장하는지를 점수화하는 알고리즘입니다. 두 가지 핵심 요소로 구성됩니다:

**TF (Term Frequency)** — 해당 문서에서 검색어가 등장하는 횟수
- "Databricks"가 5번 등장하는 문서 > 1번 등장하는 문서

**IDF (Inverse Document Frequency)** — 전체 문서에서 드물게 등장할수록 높은 점수
- "Databricks"가 100만 개 문서 중 100개에서만 등장 → 높은 IDF
- "그리고"가 100만 개 문서 중 90만 개에서 등장 → 낮은 IDF

### 비유

"삼성"이라는 단어는 너무 흔해서 검색 점수가 낮고, "갤럭시 Z 플립6"은 드물어서 검색 점수가 높습니다. **희소성이 곧 중요도**입니다.

### TF-IDF와의 관계

BM25는 TF-IDF의 **개선 버전**입니다. TF-IDF는 단어 빈도가 높을수록 점수가 무한히 올라가지만, BM25는 **포화(saturation)** 개념을 추가하여 일정 빈도 이상에서는 점수 증가가 둔화됩니다. 또한 **문서 길이 보정**이 포함되어, 긴 문서가 단순히 단어가 많다는 이유로 높은 점수를 받는 것을 방지합니다.

## 6. 하이브리드 검색 — "의미 + 키워드를 둘 다 사용"

### 왜 필요한가?

| 검색 방식 | 잘하는 것 | 못하는 것 |
|-----------|-----------|-----------|
| Dense (임베딩) | 의미적 유사성 ("자동차" → "차량" 검색 가능) | 정확한 키워드 ("ERR_404") |
| Sparse (BM25) | 정확한 키워드 매칭 | 동의어, 의미적 유사성 |

**둘 다 사용하면** 서로의 약점을 보완할 수 있습니다.

### RRF (Reciprocal Rank Fusion) — 두 결과를 합치는 방법

#### 왜 필요한가?

Dense 검색의 점수는 0~1 범위이고, Sparse(BM25) 검색의 점수는 0~수십 범위입니다. **점수의 스케일이 다르기 때문에** 단순히 더하거나 평균을 낼 수 없습니다.

#### 무엇인가?

RRF는 점수가 아닌 **순위(rank)**를 기준으로 결과를 결합합니다.

```
RRF 점수(문서 d) = Σ 1/(k + rank_i(d))

- k = 60 (기본값, 순위 차이의 영향을 조절)
- rank_i(d) = i번째 검색 방식에서 문서 d의 순위
```

**예시**:
- 문서 A: Dense 검색 1위, BM25 검색 5위 → RRF = 1/(60+1) + 1/(60+5) = 0.0318
- 문서 B: Dense 검색 3위, BM25 검색 2위 → RRF = 1/(60+3) + 1/(60+2) = 0.0321
- 결과: 문서 B가 더 높은 점수 (양쪽에서 고르게 높은 순위)

### 비유

두 명의 심사위원이 각각 독립적으로 순위를 매기고, **양쪽 심사위원 모두에게 높은 순위를 받은 후보**를 최종 선발하는 것과 같습니다.

{% hint style="info" %}
**Databricks Vector Search**는 하이브리드 검색을 기본 지원합니다. `query_type="hybrid"`로 설정하면 Dense + Sparse 검색 결과를 RRF로 자동 결합합니다.
{% endhint %}

## 7. 청킹 (Chunking) — "긴 문서를 적당한 크기로 자르기"

### 왜 필요한가?

1. **LLM의 컨텍스트 윈도우 제한**: LLM에 한 번에 넣을 수 있는 텍스트 양에 한계가 있습니다
2. **검색 정확도**: 100페이지짜리 문서 전체보다, 관련 있는 단락 하나를 찾는 것이 더 정확합니다

| 청크 크기 | 장점 | 단점 |
|-----------|------|------|
| 너무 크면 | 충분한 문맥 포함 | 노이즈가 많아 답변 품질 저하 |
| 너무 작으면 | 정확한 정보 포함 | 문맥이 부족하여 의미 손실 |

### 청킹 전략

#### Fixed-size (고정 크기)
가장 단순한 방법. 정해진 문자 수(예: 500자)로 기계적으로 자릅니다.
- **장점**: 구현이 간단, 속도가 빠름
- **단점**: 문장 중간에서 잘릴 수 있음

#### Recursive (재귀적 분할)
구분자 우선순위에 따라 자릅니다: `\n\n` (단락) → `\n` (줄바꿈) → `.` (문장) → ` ` (공백)
- **장점**: 자연스러운 경계에서 분할
- **단점**: 청크 크기가 불균일할 수 있음

#### Semantic (의미 기반)
임베딩 유사도가 **급변하는 지점**에서 자릅니다. 앞뒤 문장의 의미가 크게 달라지는 곳이 자연스러운 주제 경계입니다.
- **장점**: 가장 지능적인 분할
- **단점**: 임베딩 계산이 필요하여 느리고 비용 발생

### 오버랩 (Overlap)

청크 경계에서 정보가 잘리는 것을 방지하기 위해, 인접한 청크의 앞뒤 일부를 **겹치게** 자릅니다.

```
[==== 청크 1 ====]
              [==== 청크 2 ====]
                            [==== 청크 3 ====]
       ↑ 오버랩 영역 ↑           ↑ 오버랩 영역 ↑
```

{% hint style="info" %}
**실무 권장**: 한국어 문서에는 **Recursive 청킹 + 200~500 토큰 크기 + 50~100 토큰 오버랩**이 좋은 출발점입니다. 자세한 전략은 [청킹 전략](chunking-strategies.md) 페이지를 참고하세요.
{% endhint %}

## 8. 토크나이저 (Tokenizer) — "텍스트를 최소 단위로 쪼개기"

### 왜 필요한가?

LLM은 문자 단위가 아닌 **토큰** 단위로 텍스트를 처리합니다. 토크나이저는 텍스트를 이 토큰으로 분리하는 도구입니다.

```
영어: "Databricks is great" → ["Data", "bricks", " is", " great"]  (4 토큰)
한국어: "데이터브릭스 좋아요" → ["데이", "터", "브", "릭", "스", " 좋", "아", "요"]  (8 토큰)
```

### 주요 토크나이저 알고리즘

**BPE (Byte Pair Encoding)**: GPT 계열이 사용합니다. 자주 나오는 문자 쌍을 반복적으로 합쳐서 토큰 사전을 구축합니다.

**SentencePiece**: 다국어를 지원하며 언어에 독립적으로 작동합니다. 공백도 하나의 문자로 취급합니다.

### 한국어의 문제

영어 기반으로 학습된 토크나이저는 한글을 **바이트 단위**로 쪼개는 경우가 많습니다. 그 결과:
- 같은 의미의 텍스트도 한국어가 영어보다 **2~3배 많은 토큰**을 사용
- 토큰 비용 증가, 컨텍스트 윈도우 낭비
- 의미 단위가 아닌 임의 단위로 분리되어 임베딩 품질 저하

### Kiwi (한국어 형태소 분석기)

한국어를 **의미 단위(형태소)**로 정확히 분리하는 오픈소스 라이브러리입니다.

```
"데이터브릭스에서 벡터검색을 사용합니다"
→ ["데이터브릭스", "에서", "벡터", "검색", "을", "사용", "하", "ㅂ니다"]
   (명사)       (조사)  (명사) (명사) (조사) (동사) (동사) (어미)
```

**활용**: BM25 검색을 위한 토큰화, 청킹 시 의미 단위 분리에 사용됩니다.

### KSS (Korean Sentence Splitter)

한국어 **문장 경계**를 정확히 인식하는 라이브러리입니다. 마침표(`.`)만으로 문장을 나누면 "3.14" 같은 숫자나 "Dr." 같은 약어에서 오분리가 발생합니다. KSS는 이런 예외를 처리합니다.

**활용**: 청킹 시 문장 단위로 깔끔하게 분리할 때 사용합니다.

{% hint style="info" %}
**한국어 RAG 최적화**: 한국어 특화 토크나이저 적용은 RAG 품질에 큰 영향을 줍니다. 자세한 내용은 [한국어 RAG 최적화](korean-nlp.md) 페이지를 참고하세요.
{% endhint %}

## 9. Re-ranking — "1차 검색 결과를 다시 정렬"

### 왜 필요한가?

1차 검색(Retrieval)은 **수백만 개** 문서에서 후보를 추리는 단계이므로 속도가 중요합니다. 하지만 속도를 위해 정밀도를 희생합니다. Re-ranking은 1차 검색으로 추린 **소수의 후보**(보통 20~50개)를 더 정교한 모델로 재평가합니다.

### Bi-encoder vs Cross-encoder

| 특성 | Bi-encoder | Cross-encoder |
|------|-----------|---------------|
| 작동 방식 | 질문과 문서를 **각각 따로** 임베딩 | 질문+문서를 **함께** 입력하여 관련성 점수 산출 |
| 속도 | 빠름 (각각 한 번만 계산) | 느림 (질문-문서 쌍마다 계산) |
| 정확도 | 상대적으로 낮음 | 매우 높음 |
| 용도 | **1차 검색** (대량 문서에서 후보 추출) | **Re-ranking** (소수 후보 재정렬) |

### 비유

**Bi-encoder**는 이력서 키워드 필터링과 같습니다. 수천 장의 이력서에서 조건에 맞는 후보 20명을 빠르게 골라냅니다.

**Cross-encoder**는 면접과 같습니다. 20명의 후보와 직접 대화하며 질문에 가장 적합한 사람을 정밀하게 평가합니다.

### RAG에서의 Re-ranking 파이프라인

```
질문 → [1차 검색: Bi-encoder] → 상위 50개 문서
     → [Re-ranking: Cross-encoder] → 상위 5개 문서
     → [LLM 생성] → 최종 답변
```

{% hint style="info" %}
**실무 팁**: Re-ranking을 추가하면 검색 품질이 크게 향상되지만, 지연 시간(latency)도 증가합니다. 실시간 응답이 중요한 경우 Re-ranking 대상 문서 수를 적절히 조절하세요.
{% endhint %}

## 10. 평가 메트릭 — "RAG가 잘 작동하는지 측정"

RAG 시스템은 "검색"과 "생성" 두 단계로 나뉘므로, 각 단계를 따로 평가해야 합니다.

### Faithfulness (충실도)

> "답변이 검색된 문서의 내용**만을** 기반으로 하는가?"

LLM이 검색 결과에 없는 내용을 지어내는 것을 **환각(hallucination)**이라고 합니다. Faithfulness는 이 환각이 얼마나 적은지를 측정합니다.

- **높은 충실도**: "문서에 따르면 A 제품의 가격은 10만원입니다."
- **낮은 충실도**: "A 제품의 가격은 아마 15만원일 겁니다." (문서에 해당 내용 없음)

### Relevance (관련성)

> "검색된 문서가 질문과 관련이 있는가?"

검색 단계의 품질을 측정합니다. 아무리 LLM이 뛰어나도, 관련 없는 문서를 기반으로는 좋은 답변을 생성할 수 없습니다.

### Correctness (정확도)

> "최종 답변이 정답과 일치하는가?"

미리 준비된 정답(ground truth)과 비교하여 답변의 정확성을 측정합니다. 전체 파이프라인의 종합적인 성능 지표입니다.

### LLM-as-Judge (LLM 판정)

사람이 수백~수천 개의 답변을 하나씩 평가하는 것은 비현실적입니다. 대신 **LLM이 답변 품질을 자동으로 평가**하는 패턴입니다.

```
평가용 프롬프트 예시:

"아래 질문, 검색된 문서, 생성된 답변을 보고
 답변의 충실도를 1~5점으로 평가하세요.
 질문: {question}
 문서: {context}
 답변: {answer}"
```

{% hint style="info" %}
**Databricks 활용**: MLflow Evaluate에서 `faithfulness`, `relevance`, `correctness` 등의 평가 메트릭을 기본 제공합니다. `mlflow.evaluate()` 한 줄로 RAG 파이프라인 전체를 자동 평가할 수 있습니다. 자세한 내용은 [RAG 평가](evaluation.md) 페이지를 참고하세요.
{% endhint %}

## 11. 프롬프트 템플릿 — "LLM에게 올바르게 질문하기"

### 왜 필요한가?

LLM은 주어진 지시(프롬프트)에 따라 동작이 크게 달라집니다. RAG에서는 검색된 문서를 바탕으로 **정확하게만** 답변하도록 유도해야 합니다.

### 프롬프트의 구성 요소

```
┌─────────────────────────────────────────┐
│ System Prompt (시스템 프롬프트)            │
│ → LLM의 역할과 규칙 정의                   │
│ "당신은 고객지원 어시스턴트입니다.           │
│  제공된 문서에만 기반하여 답변하세요."       │
├─────────────────────────────────────────┤
│ Context (컨텍스트)                        │
│ → 검색된 문서 내용 삽입                     │
│ "문서 1: ... 문서 2: ... 문서 3: ..."      │
├─────────────────────────────────────────┤
│ Question (사용자 질문)                     │
│ → 사용자의 원래 질문                        │
│ "Databricks Vector Search 사용법은?"       │
└─────────────────────────────────────────┘
```

### 좋은 RAG 프롬프트의 핵심 원칙

1. **근거 한정**: "제공된 문서에**만** 기반하여 답변하세요"
2. **불확실성 인정**: "문서에 답이 없으면 **모른다고** 말하세요"
3. **출처 명시**: "답변의 근거가 되는 **문서 번호를 인용**하세요"
4. **형식 지정**: 답변 형식(마크다운, 번호 목록 등)을 구체적으로 지시

{% hint style="info" %}
**핵심**: RAG 프롬프트에서 가장 중요한 지시는 **"모르면 모른다고 하세요"**입니다. 이 한 줄이 환각(hallucination)을 크게 줄여줍니다. 자세한 내용은 [RAG 체인 구축](chain-building.md) 페이지를 참고하세요.
{% endhint %}
